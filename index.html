Overview

Project Overview

Search Reasoning State Management Evidence Extraction Skill Reuse Error Oversight Coverage-Driven

SearchOS is operating-system-level infrastructure for search agents. It lifts the recurring problems of deep search — context governance, shared state, anomaly oversight, knowledge reuse, report synthesis — into system-level services, so search agents run like scheduled, supervised processes.

Its reference implementation pairs a long-lived Orchestrator — an LLM controller that plans, dispatches, and converges from real-time SOCM snapshots — with short-lived sub-agents, all wrapped by middleware that runs like a kernel in the background.

Problem Formulation · Open-Domain Information-Seeking Task

A query q decomposes into a chain of sub-question nodes; each table Tm (an entity×attribute coverage grid) frames the nodes it grounds. Two shapes recur: tables partition the chain to cover breadth, or tables overlap so that the node(s) framed by all of them form the answer set satisfying every constraint.

Agent Lifecycle · Multi-Agent Scheduling Timeline

A worked case: q = “Compare leading 2025 LLMs — context window, price, benchmark”. Swimlanes per agent, time flows left→right. Solid arrows = dispatch / trigger (control); dashed arrows = return result.

Core Capabilities

  • Automatic schema extraction (entity × attribute)
  • Coverage-driven exploration priority
  • Evidence Graph auto-extraction
  • Anti-Skill mining from failure trajectories

Three Sub Agent Types

  • explore_agent — Disambiguation / candidate enumeration
  • search_agent — Evidence collection
  • writer_agent — Structured drafting / finalize

Conflict adjudication is handled at the system layer (Scheduler + conflict arbitration), not by a dedicated agent.

Architecture

Architecture Design

Three parts make the diagram work: the agents that decide, the state they share, and the middleware that runs around every step.

1

Multi-Agent System

One brain, a small crew. A single Orchestrator plans and decides; it never searches the web itself — it hands out jobs to short-lived sub-agents and checks their work.

Explore Agent the scout

When a question is vague, it does a quick reconnaissance pass — what's out there, which entities exist, where the ambiguities are — so the Orchestrator can draft a sensible plan. It writes no answer.

Tools: search / open / find only, no evidence extraction. Optional (enable_explore); spawned via enqueue_tasks(agent_type="explore_agent") only when the schema can't be drafted directly. Persona in explore_agent/agent.md.

Search Agent the investigator

The workhorse. Given a few specific cells to fill, it searches, opens authoritative pages, and pulls out facts. Several run in parallel, each on its own slice of the problem.

Tools: browser (search/open/find) + access-skill executors. Findings reach state automatically — middleware extracts evidence from its open/find results; the agent itself never writes SOCM. Batched via enqueue_tasks. Persona in search_agent/agent.md.

Writer Agent the author

Once enough evidence is in, one long-lived writer reads the collected facts and drafts the report section by section — citing sources, flagging conflicts, and pointing out remaining gaps. It never browses the web.

Tools: outline tools (write_section with cited evidence ids…) + SOCM read tools (read_coverage/read_evidence…); no browser. Singleton per session, gated by enable_writer_agent. Triggered by WriterTriggerSensor at a coverage stage (default ≥ 0.5) or proactively woken via send_followup. Persona in writer_agent/agent.md.

The Orchestrator is the sole decision-maker — it reads the latest state each step and decides what to dispatch next or when to converge. Each sub-agent runs on a frozen state snapshot, so a dozen work in parallel without their reasoning bleeding together. Tap a card for the real tools & triggers.

2

State Model

The shared memory. Everything the agents learn is folded into four living objects — not a growing chat log.

Frontier the to-do list

A priority queue of search tasks. Each task moves pending → running → done (or blocked / cancelled), so the system always knows what's left to explore.

FrontierMemory of FrontierTask. Orchestrator pushes via enqueue_tasks; the Scheduler dispatches, unblocks, and reaps.

Evidence Graph the facts

Every claim found on the web becomes a node, linked by support / conflict / refine edges — so contradictions surface instead of getting buried.

EvidenceGraph. Written automatically by EvidenceExtractionMiddleware after each tool call.

CoverageMap the progress sheet

An entity × attribute table that marks each cell missing / filled / uncertain / hard. It's the system's sense of "what do we still not know?"

CoverageMap of CoverageCell (CellStatus). Seeded by create_schema(), backfilled as evidence grows; drives gap detection and Writer structure.

Strategy Log the lessons

A record of what isn't working — repeated queries, dead-end sources — so the system stops digging in the same dry hole.

StrategyMemory (state attr strategy_log): StrategyPattern anti-patterns + FailureMemory. Feeds the Sensor layer and post-mortem distillation.

All four are persisted to workspace/<session>/search_state.json with atomic file locking, so multiple sub-agents can update them in parallel without clobbering each other. Tap a card for the real class names.

3

Middleware

The automatic plumbing. Hooks wrap every model call and tool call, so the agents stay focused while context, safety, and bookkeeping happen on their own.

Control before the LLM thinks

Keeps the prompt small and cache-friendly: layer the context, trim overflow, and splice in a skill playbook when one applies.

LayeredContextMiddleware · DynamicTrimMiddleware · ControlMiddleware (skill injection). Bound to awrap_model_call.

Sensor after a tool returns

Watches for trouble — loops, repeated queries, stalls — and either nudges the agent with advice or hard-stops runaway dispatch. Also fires the writer trigger.

Rule-based: LoopSensorImpl, StallEscalationSensor, DispatchRoundSensor, WriterTriggerSensor. LLM-based: StrategySensorImpl, EvaluatorSensorImpl (4-D verdict). Bound to awrap_tool_call / checkpoints.

Extraction turning pages into facts

When a page is read, it cleans and de-dupes the text, runs an LLM judge to pull out claims, and writes them straight into the Evidence Graph and CoverageMap.

EvidenceExtractionMiddleware: gating (schema / length / error / dup-hash) → LLM extract & merge-by-URL → persist EvidenceNode + provenance → Coverage backfill. Background flush, synchronous drain on the final round.

Two hook points hold it together. awrap_model_call shapes context going in and reads reasoning coming out; awrap_tool_call guards budget, runs the sensors, and triggers extraction. Sub-agents only read pages — middleware is the single place that writes state. Tap a card for the real class names.

Middleware is mounted on model_call and tool_call events; EvidenceExtraction writes evidence when tool results or final summaries are extractable.

Case

Case Study

Follow one question on its journey — from the moment it arrives to the finished report.md. Five milestones, one of them a loop.

1

Question In entry

A question arrives. SearchOS spins up a fresh workspace, starts with a blank slate of state, and wakes the Orchestrator — the single mind that will run the whole search.

2

Plan & Map decompose

The Orchestrator turns the question into a plan. When things are vague it sends a scout ahead first; then it draws a coverage map — a table of exactly which facts still need to be found.

3

Search Loop repeats

The heart of it. Several search agents fan out in parallel, read pages, and evidence is captured automatically into the graph and the map. The Orchestrator checks progress and keeps dispatching — round after round — until coverage is enough.

4

Write Up synthesize

Once the map is full enough, the writer agent reads the collected evidence and drafts the answer section by section — every claim tied back to its sources, conflicts flagged rather than hidden.

5

Report Out deliver

The harness assembles the final document and always guarantees a report.md — falling back through simpler synthesis modes if needed, so a session never ends empty-handed.

Along the way, dead-end and low-yield searches are quietly distilled into a failure memory — so the system, and future sessions, stop repeating the same mistakes.
